# SmartRevision architecture

SmartRevision is an adaptive learning companion: a React 18 + Vite PWA
(Vercel) over Supabase, organised as four small layers with strict import
rules. This document is the map; the module headers are the details.

```
screens/  ──►  components/, lib/  ──►  engine/, curriculum/, data/
                                        │           │            │
                                     pure logic  bundled CKB   supabase
```

## Layers

### `src/engine/` — pure domain logic
Pure functions only: **no React, no supabase, no fetch, no clock reads beyond
`now` parameters**. Everything here is unit-tested, and product decisions are
pinned in the tests (see `forecast.test.js`, `mission.test.js`).

| Module | Owns |
|---|---|
| `schedule.js` | the 0/1/7/30/120 interval table, exam-date truncation, revision-row building, adaptive rescheduling (`rescheduleAfter`) |
| `metrics.js` | due/missed buckets, the forgetting curve (`computeMemory`), dashboard rollups |
| `forecast.js` | memory-at-exam simulation, the Home forecast card states |
| `scoring.js` | the one importance formula (blueprint weight × type × weakness × PYQ) shared by daily and long-horizon planning |
| `mission.js` | **Adaptive Planning Engine**: `buildMission(availableMin, …)` → Today's Mission; the coach-mode gate (`coachModeEnabled`) |
| `recovery.js` | **Momentum Engine**: recovery queue, flexible revision windows, Memory Health |
| `roadmap.js` | Active Chapter rules: progress, advance *suggestion* at 85% |
| `receipt.js`, `notifyCopy.js` | exam receipt stats, notification copy |
| `types.js` | JSDoc typedefs for the shared shapes |

### `src/curriculum/` — Curriculum Knowledge Base (CKB)
Static, exam-agnostic content bundled with the app. **No student state, ever.**
`data/` is generated by `npm run build:curriculum` from the transcription
sources in `scripts/curriculum-src/`; never edit the JSON by hand.

- `ckb.js` — manifest + lazy per-subject loading (each subject file is its own
  Vite chunk; a phone downloads only the subjects it uses).
- `blueprints.js` — per-exam chapter weights (CBSE 11/12, GSEB 12, JEE Main).
  Content and exam pricing are separate on purpose: the same chapter tree
  serves every board; each exam prices it in its own blueprint file.
- **Id rule:** chapter/topic ids (`p1`, `c11_3`, `p1.5`, …) are stable forever
  and append-only — student rows reference them via `curriculum_topic_id`.
  `curriculum.test.js` + `id-snapshot.json` enforce this; the build script
  refuses to drop a published id.

`src/lib/syllabus.js` is a synchronous adapter over the same JSON for the
lazily-loaded syllabus screens (picker/calendar/settings); always-on surfaces
use the async `ckb.js` loaders.

### `src/data/` — repositories
All `supabase.from()` access for the domain tables. Screens **never** query
these tables directly in new code — this layer is the seam where a local-first
store (IndexedDB + sync) can slot in later without the engines noticing.
`revisionsRepo`, `topicsRepo`, `planItemsRepo`, `missionsRepo`.

Integrity-sensitive logic (gems, streak, streak-freeze, pro limits) stays in
Postgres functions/triggers behind RLS — the client only reads the results.

### `src/screens/`, `src/components/`, `src/lib/`
Screens are routes; `components/` holds extracted shared components
(`MissionCard`); `lib/` holds contexts, hooks and shared UI. The three
syllabus screens are lazy routes so the curriculum JSON stays out of the main
bundle.

## The two modes

- **Mode 1 — Guided (Classes 1–10):** the original flow, untouched. Students
  pick what to study; the app tracks revisions, streak, memory.
- **Mode 2 — Intelligent Study Coach (Classes 11–12):** `coachModeEnabled`
  gates a `MissionCard` on Home. The student says how long they have; the
  mission engine proposes revisions (≤40% of the day) + recovery (≤20%) + new
  learning from each subject's Active Chapter, weighted by the exam blueprint.
  Accept persists one mission per day; regenerate is deterministic by seed.

Both modes share every other system. With no roadmap/CKB link the mission
degrades to revisions+recovery, so custom and scanned content lose nothing.

## No-guilt invariants (do not regress)

- Never render an overdue/backlog count in coach mode — Memory Health only.
- Missed revisions return through the capped recovery slice, gradually.
- Revision dates are recommendations with flexible windows (±35%), not
  deadlines; a late completion reschedules the rest from the actual date.
- Truncate at the exam date, never compress spacing.
- The planner runs entirely on-device from bundled data. AI (scan, subtopic
  suggestions) only ever produces *inputs* to the planner, never decisions.

## Database

Migrations live in `supabase/migrations/` as ordered numbered files (see its
README; they are run by hand in the SQL editor). New DDL = new file, never an
edit. The service worker (`public/sw.js`) is network-first for navigations —
online always loads the latest deploy; the cached shell is offline-only.
